# Yakudoc — 작업 프롬프트 템플릿 (Claude 작성용)

> 이 문서는 **Claude(프롬프트 작성자)** 가 **Codex 작업 에이전트** 에게 넘길 작업 지시 프롬프트를
> 일정한 형식으로 작성하기 위한 규격이다. 모든 고도화 작업 프롬프트는 아래 골격을 그대로 따른다.
>
> 역할 구분: Claude는 프롬프트 작성 + 결과 독립 검증을 맡고, 실제 코딩은 Codex가 한다.
> 따라서 프롬프트는 **한 번에 하나의 완결된 작업 단위**로, 검증 가능한 완료 조건을 담아야 한다.

---

## 1. 작성 원칙

- **작업 근거는 항상 문서로 지목한다.** 임의 판단 여지를 남기지 않는다.
  (예: `advanced_spec.md 4장`, `architecture.md 11장`)
- **한 프롬프트 = 한 작업 단위.** 여러 기능을 한 번에 넘기지 않는다. 권장 순서는
  `advanced_spec.md 10장(구현 권장 순서)`을 따른다.
- **문서→마이그레이션→코드** 순서를 프롬프트 안에서도 단계로 나눈다.
- **범위를 명시적으로 닫는다.** 대상 파일 외 수정 금지, 신규 파일은 목록으로 못박는다.
- **완료 조건은 검증 가능하게.** "동작한다" 대신 "이 요청에 이 응답/상태코드가 온다"로 쓴다.
- 하네스/AGENTS 규칙은 매번 반복 서술하지 않고 **참조로 상속**시킨다(아래 §2 상단 고정 문구).

---

## 2. 프롬프트 골격

아래 블록을 복사해 채운다. 대괄호 `[...]`는 채울 자리다.

```
# 작업: [작업명]

## 전제
- 시작 전 AGENTS.md의 필독 3문서를 읽고 규칙을 준수한다.
- 근거 문서: [advanced_spec.md N장 / architecture.md N장]
- 구현 권장 순서상 이번 작업은 [N번째 / 선행 작업: ...] 이다.

## 배경 / 목적
[이 작업이 무엇을, 왜 하는지 2~3줄. 사용자 관점의 결과.]

## 대상 파일
- 수정: [경로1, 경로2 ...]
- 신규: [경로a, 경로b ...]   ← 없으면 "없음"
- 그 외 파일은 수정하지 않는다.

## 구현 지시
1. (문서) [설계 변경이 있으면 근거 문서 갱신 — 없으면 생략]
2. (마이그레이션) [migrate.js에 추가할 DDL. advanced_spec의 DDL 스니펫을 그대로 지목]
3. (백엔드) [컨트롤러/라우트/서비스 변경 지시. 엔드포인트·권한·에러코드 명시]
4. (프론트) [화면/호출 변경 지시 — 해당 시]

## 제약
- AGENTS.md 1~2장 준수(설계 준수/수정 범위/무한 루프 방지/재시도 규칙).
- 설계와 다르게 해야 할 이유가 생기면 임의 변경 없이 즉시 보고하고 확인받는다.
- [작업 특수 제약: 예) 원본 미저장, 기존 API 응답 형식 불변 등]

## 완료 조건 (검증 가능 형태)
- [ ] [요청 예시 → 기대 응답/상태코드]
- [ ] [권한 분기: 예) lead 아닌 사용자는 403]
- [ ] [AGENTS.md 3장 품질 체크리스트 중 해당 항목]

## 보고
- AGENTS.md 4장 보고 형식([완료]/[에러])으로 결과를 보고한다.
```

## 3. 작성 예시 — 기능 2 (업체 owner 이양)

```
# 작업: 업체 owner(담당자) 이양 기능 구현

## 전제
- 시작 전 AGENTS.md의 필독 3문서를 읽고 규칙을 준수한다.
- 근거 문서: advanced_spec.md 4장, architecture.md 11장(공통 에러 응답)
- 구현 권장 순서상 이번 작업은 1번째(스키마 변경 최소, 단독 완결)이다.

## 배경 / 목적
담당자 퇴사/이직 시 업체 owner 권한을 다른 사용자에게 이양한다.
기존 owner는 접근이 끊기지 않도록 편집 권한 멤버로 남길 수 있어야 한다.

## 대상 파일
- 수정: backend/src/controllers/companies.js, backend/src/routes/companies.js
- 신규: 없음
- 그 외 파일은 수정하지 않는다.

## 구현 지시
1. (문서) 설계 변경 없음 — advanced_spec 4장 그대로 구현.
2. (마이그레이션) 스키마 변경 없음(company_history는 기능 3에서 도입).
   이번 작업은 이력 기록 없이 owner_id 이양만 구현하고, 이력 연계는 기능 3에서 추가한다.
3. (백엔드) POST /api/companies/:id/transfer-owner 구현.
   - Request: { new_owner_id, keep_as_member?=true }
   - 권한: 현재 owner 또는 admin(아니면 403)
   - new_owner 없음/비활성 → 404, 자기 자신 이양 → 400
   - 트랜잭션: owner_id 갱신 → keep_as_member면 기존 owner를 company_members(edit)로 INSERT
   - Response 200: { id, owner_id }

## 제약
- AGENTS.md 1~2장 준수.
- 기존 companies API 응답 형식은 변경하지 않는다.

## 완료 조건 (검증 가능 형태)
- [ ] owner가 호출 시 200 + owner_id가 new_owner로 바뀜
- [ ] 제3자(owner/admin 아님) 호출 시 403
- [ ] 존재하지 않는 new_owner_id → 404, 자기 자신 이양 → 400
- [ ] keep_as_member=true일 때 기존 owner가 company_members(edit)로 남음
- [ ] 트랜잭션 실패 시 owner_id/멤버가 롤백됨

## 보고
- AGENTS.md 4장 보고 형식으로 결과를 보고한다.
```

## 4. Claude 발송 전 셀프 체크

프롬프트를 Codex에 넘기기 전에 스스로 확인한다.

- [ ] 근거 문서와 절 번호를 정확히 지목했는가
- [ ] 대상 파일(수정/신규)을 닫힌 목록으로 명시했는가
- [ ] 엔드포인트·권한·상태코드가 명세와 일치하는가
- [ ] 완료 조건이 "검증 가능"한 형태인가(요청→응답/상태코드)
- [ ] 한 프롬프트가 한 작업 단위인가(과대 범위 아닌가)
- [ ] 특수 제약(원본 미저장, 응답 형식 불변 등)을 빠뜨리지 않았는가

## 5. Claude 결과 검증 시 관점

Codex 결과를 독립 검증할 때는 아래를 직접 확인한다(에이전트 보고를 그대로 신뢰하지 않는다).

- 실제 diff가 대상 파일 범위 안에 있는가(범위 외 변경 여부)
- 마이그레이션이 idempotent하고 기존 데이터와 하위호환인가
- 완료 조건의 각 항목을 실제 요청/쿼리로 재현했는가
- 권한 분기(owner/lead/admin, personal/team)가 실제로 막히는가
- 원본 미저장 등 보안 제약이 코드에서 지켜지는가
